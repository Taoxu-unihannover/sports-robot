# 开源球类机器人项目吸收与超越计划

> 创建日期：2026-06-09  
> 目标：拿到任意球类机器人开源项目 A 后，sports-robot 能自动拆解其技术栈，沉淀为标准 skills/plugins；再基于 sports-robot 自身能力重现项目 A，达到同等性能；随后横向比较 A 的方法与 sports-robot 已有同类方法，选择各技术栈最优方法组合，最终形成性能超过项目 A 的新方案，并输出完整对比报告。

## 一、总体功能定义

本功能建议实现为一个主 plugin 加三个核心 skills：

- 主 plugin：`ball-project-assimilator`
- Skill 1：`open-project-skill-distiller`
- Skill 2：`stack-method-benchmark`
- Skill 3：`best-stack-composer`

三者关系：

```text
开源项目 A
  -> open-project-skill-distiller
  -> 项目 A 技术栈拆解 + 新 skills/recipes
  -> sports-robot 复现项目 A
  -> stack-method-benchmark
  -> A 的方法 vs sports-robot 已有方法横向对比
  -> best-stack-composer
  -> 最优技术栈组合
  -> 超越 A 的增强项目 + 总体性能报告
```

## 二、核心工作流

### 阶段 0：输入与约束声明

输入：

- 开源项目 A 的本地路径或仓库地址。
- 项目类型：网球、乒乓球、羽毛球、棒球、足球等。
- 复现目标：仿真、感知、控制、击球、移动底盘、机械臂、整机系统等。
- 性能指标：成功率、命中率、轨迹误差、推理延迟、训练吞吐、episode reward、真实/仿真一致性等。
- 允许使用资源：是否允许下载模型/数据集，是否允许 GPU 长训练，是否允许使用项目 A 的资产。

硬性约束：

- sports-robot 可以分析项目 A 的技术栈、接口、配置和算法思路。
- 不直接复制项目 A 的业务代码作为 sports-robot 标准能力。
- 如果需要使用项目 A 的专有 mesh、模型权重、数据集，必须在报告中单独标注许可证和依赖风险。
- 复现项目必须通过 sports-robot 的 skills/plugins 生成。

### 阶段 1：拆解项目 A 并沉淀为 skills

目标：把项目 A 的技术栈拆成 sports-robot 可复用能力。

执行步骤：

1. 扫描项目 A：
   - 识别语言、依赖、环境、入口脚本、训练脚本、测试脚本。
   - 识别仿真平台：MuJoCo、Isaac Sim、PyBullet、Gazebo、ROS2、自研仿真等。
   - 识别感知栈：检测、跟踪、状态估计、相机标定、图像/点云处理。
   - 识别建模栈：球动力学、反弹、旋转、空气阻力、接触模型。
   - 识别控制栈：MPC、IK、RL、IL、QP、轨迹优化、技能库。
   - 识别执行栈：机械臂、移动底盘、发球机、伺服驱动、安全边界。
   - 识别工程栈：训练框架、日志、配置、评估、可视化、CI。

2. 生成技术栈图谱：
   - 输出 `project-a-stack-map.md`。
   - 每个模块标明输入、输出、依赖、指标、可替代 sports-robot skill。

3. 匹配 sports-robot 已有 skills：
   - 已覆盖：记录 skill 名称与覆盖比例。
   - 部分覆盖：记录需要补充的 schema、脚本、recipe。
   - 未覆盖：生成新 skill 候选。

4. 沉淀新 skills：
   - 每个新 skill 使用标准结构：`SKILL.md`、`scripts/`、`references/`、`recipes/`、`tests/`。
   - skill 描述必须通用化，不能只服务项目 A。
   - 每个 skill 至少有 smoke test。

5. 沉淀 plugin recipe：
   - 为项目 A 生成复现 recipe：`plugins/ball-project-assimilator/recipes/project-a-reproduction/RECIPE.md`。
   - recipe 记录调用哪些 skills、顺序、配置、验证命令和指标。

验收标准：

- 项目 A 的关键技术栈覆盖率达到 100%。
- 未覆盖项全部转成 skill/plugin backlog。
- 至少能通过 sports-robot 生成项目 A 的可运行复现骨架。

### 阶段 2：用 sports-robot 重现项目 A

目标：sports-robot 只通过自身 plugins/skills 开发或重现项目 A，初步达到项目 A 相同性能效果。

执行步骤：

1. 创建复现项目目录：
   - 建议命名：`reproductions/project-a/` 或用户指定目录。

2. 通过 plugin 生成工程：
   - 环境配置。
   - 仿真场景。
   - 感知模块。
   - 建模模块。
   - 控制/训练模块。
   - 评估与可视化模块。
   - 测试与报告脚本。

3. 运行可运行性验证：
   - 导入测试。
   - 仿真加载测试。
   - 环境 `check_env`。
   - 短训练 smoke test。
   - 评估脚本 smoke test。
   - 可视化导出测试。

4. 运行性能复现：
   - 使用与项目 A 尽量一致的数据、环境、随机种子、训练步数、硬件条件。
   - 记录性能指标。

5. 输出复现报告：
   - `project-a-reproduction-report.md`
   - 包括复现成功项、失败项、性能差距、缺失能力、下一轮 skill 重设计任务。

通过标准：

- 可运行性 100% 通过。
- 核心性能达到项目 A baseline 的 95% 以上。
- 若项目 A baseline 无法复现，先冻结可复现 baseline，再继续。

失败处理：

- 如果性能低于 baseline 95%，不能只在复现项目内修补。
- 必须归因到 sports-robot 的某个 skill/plugin。
- 重设计该 skill/plugin 后重新生成或更新复现项目。

### 阶段 3：横向对比同类技术栈

目标：将项目 A 的各技术栈方法，与 sports-robot 已有类似技术栈横向比较，找出同一技术栈中更优的方法或 skill。

对比维度：

| 技术栈 | 典型候选 | 指标 |
|---|---|---|
| 球检测 | 项目 A detector、YOLO、HSV、ONNX、小目标检测模型 | mAP、召回率、延迟、鲁棒性 |
| 球跟踪 | 项目 A tracker、滑动窗口、Kalman、TrackNet 风格模型 | 轨迹连续性、丢帧恢复、延迟 |
| 状态估计 | 项目 A estimator、CV/CA/EKF/UKF/粒子滤波 | 位置 RMSE、速度 RMSE、稳定性 |
| 球动力学 | 项目 A dynamics、抛物线、阻力、Magnus、数据驱动模型 | 预测误差、泛化、计算成本 |
| 反弹/接触 | 项目 A contact、解析模型、学习模型、混合模型 | 反弹点误差、速度误差 |
| 规划 | 项目 A planner、MPC、轨迹优化、采样规划 | 命中率、可达率、计算延迟 |
| 控制 | 项目 A controller、MPC、QP、RL、IL、残差控制 | 稳定性、误差、实时性 |
| RL 训练 | 项目 A trainer、SB3、RLlib、自研 runner | 收敛速度、最终性能、吞吐 |
| 仿真 | 项目 A simulator、MuJoCo、Isaac、PyBullet | 物理可信度、速度、可维护性 |
| 可视化/评估 | 项目 A tools、sports-robot evaluator/web viewer | 指标完整度、可重复性 |

执行步骤：

1. 为每个技术栈建立统一 benchmark harness。
2. 将项目 A 的方法和 sports-robot 的同类 skills 接入同一输入输出 schema。
3. 使用相同数据、相同随机种子、相同任务配置运行评测。
4. 输出单项排名。
5. 标记最优方法、次优方法、适用场景和失败模式。

输出报告：

- `project-a-stack-benchmark-report.md`
- 每个技术栈一张表：
  - 方法名称
  - 所属 skill/plugin
  - 性能指标
  - 资源消耗
  - 优缺点
  - 是否进入最优组合

通过标准：

- 每个关键技术栈至少比较项目 A 方法和一个 sports-robot 替代方法。
- 所有对比使用同一评测入口和同一指标定义。
- 报告明确说明“哪个方法更好、为什么更好、代价是什么”。

### 阶段 4：组合最优方法并超过项目 A

目标：组合阶段 3 中各技术栈最优方法，形成一整套更优系统，性能超过项目 A。

执行步骤：

1. 生成最优组合配置：
   - `best_stack.yaml`
   - 记录每个技术栈选择的 skill、方法、版本、参数。

2. 检查组合兼容性：
   - 输入输出 schema 是否一致。
   - 控制频率是否匹配。
   - 延迟是否满足闭环需求。
   - 坐标系、单位、时间戳是否一致。
   - 安全边界是否保留。

3. 生成增强项目：
   - 建议命名：`enhanced/project-a-plus/` 或用户指定目录。

4. 运行整体评测：
   - 与项目 A baseline 比较。
   - 与 sports-robot 复现版比较。
   - 与各单项最优但未组合版本比较。

5. 输出整体性能对比报告：
   - `project-a-plus-performance-report.md`
   - 说明是否超过项目 A、超过多少、在哪些指标上超过、是否有指标退化。

超过标准：

- 核心任务指标超过项目 A baseline 至少 5%。
- 不允许安全性、稳定性、可运行性退化。
- 允许部分非核心指标小幅下降，但必须在报告中解释代价。

建议核心指标判定：

| 类型 | 超越标准 |
|---|---|
| 成功率/命中率 | 高于项目 A 至少 5% |
| 平均 reward | 高于项目 A 至少 5% |
| 轨迹/落点误差 | 低于项目 A 至少 5% |
| 推理延迟 | 不高于项目 A 105%，目标低于项目 A |
| 鲁棒性 | 噪声/扰动场景中高于项目 A 至少 5% |
| 可复现性 | 固定随机种子下结果方差不高于项目 A |

## 三、主 plugin 设计：`ball-project-assimilator`

### 3.1 plugin 职责

`ball-project-assimilator` 是项目吸收、复现、横向评测、最优组合的总控插件。

核心能力：

- 扫描任意球类机器人项目。
- 生成技术栈图谱。
- 将项目能力转为 sports-robot skills/recipes。
- 使用 sports-robot 自举复现项目。

## 四、核心 skills 设计

### 4.1 `open-project-skill-distiller`

用途：把开源项目 A 的技术栈拆解并转成 sports-robot 标准 skills。

输入：

- `source_project_path`
- `target_domain`
- `license_policy`
- `allow_asset_reuse`
- `target_skills_dir`
- `target_plugins_dir`

输出：

- `stack_map`
- `skill_gap_report`
- `generated_skill_specs`
- `generated_recipes`
- `reproduction_plan`

SKILL.md 要点：

- 先扫描项目结构，再读入口文件。
- 先抽象接口、数据流、指标，再写 skill。
- 每个 skill 必须有触发描述、输入输出、执行步骤、失败处理、验证方式。
- 对项目 A 的实现只做概念沉淀，不做业务源码复制。
- 对许可证敏感资源输出风险标记。

### 4.2 `stack-method-benchmark`

用途：横向比较同一个技术栈中项目 A 方法与 sports-robot 已有方法的性能。

输入：

- `stack_map`
- `candidate_methods`
- `benchmark_dataset_or_env`
- `metric_schema`
- `random_seed_set`

输出：

- `stack_benchmark_report`
- `method_ranking`
- `best_method_per_stack`
- `tradeoff_notes`

SKILL.md 要点：

- 所有候选方法必须适配同一 schema。
- 所有评测必须固定随机种子和指标定义。
- 对延迟、精度、稳定性、资源消耗同时记录。
- 报告必须说明最优方法的适用范围，不只给排名。

### 4.3 `best-stack-composer`

用途：把每个技术栈的最优方法组合成整体系统，并验证能否超过项目 A。

输入：

- `best_method_per_stack`
- `compatibility_schema`
- `baseline_metrics`
- `target_improvement`

输出：

- `best_stack.yaml`
- `enhanced_project_plan`
- `compatibility_report`
- `plus_performance_report`

SKILL.md 要点：

- 组合前必须检查 schema、频率、延迟、坐标系、单位、安全约束。
- 单项最优不等于整体最优，若组合后退化，要记录冲突原因。
- 允许选择次优方法替代单项最优，只要整体性能更好。
- 最终报告必须同时对比项目 A、sports-robot 复现版、增强版。

## 五、循环机制与退出机制

### 5.1 三层循环

本功能有三层循环：

```text
第一层：技能沉淀循环
项目 A -> 拆解 -> skills -> 复现失败 -> 补 skill -> 再复现

第二层：横向评测循环
同一技术栈候选方法 -> benchmark -> 指标不可信 -> 修 benchmark -> 再评测

第三层：最优组合循环
单项最优组合 -> 整体评测 -> 性能未超越 A -> 分析冲突 -> 换组合/补 skill -> 再评测
```

### 5.2 进入下一阶段条件

| 当前阶段 | 进入下一阶段条件 |
|---|---|
| 技术栈沉淀 | 项目 A 关键技术栈覆盖率 100%，复现项目可运行性 100% |
| 项目复现 | 核心性能达到项目 A baseline 95% 以上 |
| 横向评测 | 每个关键技术栈至少完成 A 方法与 sports-robot 方法的同基准对比 |
| 最优组合 | 组合方案通过兼容性检查 |
| 超越验证 | 增强版核心指标超过项目 A 至少 5% |

### 5.3 退出条件

成功退出：

- 复现版达到项目 A baseline 95% 以上，并输出复现报告。
- 增强版核心指标超过项目 A baseline 5% 以上，并输出总体性能对比报告。

带差距退出：

- 复现版达标，但增强版未超过项目 A；输出瓶颈报告和下一步优化 backlog。
- 某些单项指标超过 A，但整体系统未超过 A；输出组合冲突报告。

失败退出：

- 同一阶段连续 3 轮失败，且失败原因相同。
- 总迭代轮数达到 5 轮仍无法达到复现版 baseline 95%。
- 项目 A baseline 无法复现，且 2 轮内无法建立替代 baseline。
- 需要外部数据集、专有模型、大规模 GPU 长训练或真实硬件，且未获得人工批准。

## 六、报告体系

每个项目 A 至少生成四类报告：

| 报告 | 路径建议 | 内容 |
|---|---|---|
| 技术栈拆解报告 | `docs/todo/project-a-stack-map.md` | 项目结构、模块、依赖、数据流、指标、skill 映射 |
| 复现报告 | `docs/todo/project-a-reproduction-report.md` | sports-robot 复现过程、指标、差距、缺口 |
| 横向评测报告 | `docs/todo/project-a-stack-benchmark-report.md` | 每个技术栈中不同方法/skill 的性能对比 |
| 超越报告 | `docs/todo/project-a-plus-performance-report.md` | 最优组合方案与项目 A、复现版的整体性能对比 |

报告必须包含：

- 评测日期、硬件、软件版本。
- baseline 来源与命令。
- 随机种子。
- 数据集或仿真配置。
- 指标定义。
- 原始结果表。
- 结论与下一步。

#### 关键发现：观测向量重建中的速度来源问题

在分析 `env._get_obs()` 与 `_compute_derived_dims()` 的差异时，发现了一个关键问题：

**`env._get_obs()` 内部使用了两种不同的速度来源：**

| 观测维度 | 速度来源 | 说明 |
|---|---|---|
| dims 10-12 (ball_vel) | 位置差分: `(pos_t - pos_{t-1}) / dt` | 球在 base 坐标系下的速度 |
| dim 13 (TTC) | `self.data.qvel[0:2]` | 球在世界坐标系下的 MuJoCo 物理速度 |

这两种速度在数值上有显著差异，因为位置差分速度包含机器人运动的影响。

#### Coriolis 修正：从 base 速度恢复世界速度

在 real 模式下，我们无法直接获取 `qvel[0:2]`，必须从 base 坐标系速度推算球的世界坐标系速度。正确的转换公式需要包含 Coriolis 项：

```
v_ball_world = v_robot_world + R_b2w @ v_ball_base + ω × r_world
```

其中：
- `v_robot_world` = 机器人全局速度 (qvel[6:8])
- `R_b2w @ v_ball_base` = 球在 base 坐标系速度旋转到世界坐标系
- `ω × r_world` = Coriolis 修正项，`ω = robot_wz`，`r_world` = 球相对机器人的世界坐标位置

**Coriolis 修正验证结果（20 episodes）：**

| 速度来源 | TTC 误差 (vs qvel) | 成功率 |
|---|---|---|
| `R_b2w @ ball_vel_base` (无修正) | 0.012~0.973 m/s | **5%** |
| `v_robot + R_b2w @ ball_vel_base` (无Coriolis) | 0.005~0.299 m/s | **5%** |
| `v_robot + R_b2w @ ball_vel_base + ω×r` (Coriolis修正) | 0.000~0.009 m/s | **95%** |
| `qvel[0:2]` (仿真真值) | 0 | **95%** |

**结论：Coriolis 修正使 TTC 计算精度提升 2 个数量级，成功率从 5% 跃升至 95%。**

#### Kalman 滤波器速度估计问题

ObjectKalmanFilter 的速度估计在球高速运动时收敛极慢：

| 指标 | Kalman 速度估计 | 真实速度 |
|---|---|---|
| 速度均值 | ~3.7 m/s | ~13.5 m/s |
| 速度误差均值 | 12.3 m/s | - |
| 速度误差最大值 | 18.2 m/s | - |

**原因**：`process_noise=0.05` 和 `measurement_noise=0.08` 的参数设置使得 Kalman 滤波器对速度的收敛时间常数过长（约 50-100 步），而接球 episode 仅持续 300-400 步。

**解决方案**：使用位置差分计算球速度（与 `env._get_obs()` 一致），Kalman 滤波器仅用于位置平滑和时延补偿。

### 10.9 关键技术洞察

#### 洞察 1：观测向量的双速度源设计

`env._get_obs()` 的 16 维观测向量内部使用了两种不同的速度来源，这是一个容易被忽略但影响巨大的设计细节：

- **dims 10-12**：位置差分速度 `(pos_t - pos_{t-1}) / dt`，用于策略网络直接消费
- **dim 13 (TTC)**：MuJoCo qvel 物理速度，用于计算碰撞时间

这种设计意味着在 real 模式下，观测重建必须同时正确处理两种速度来源。简单地用 Kalman 速度同时替代两者会导致 TTC 计算严重失真。

#### 洞察 2：旋转坐标系的速度变换需要 Coriolis 修正

从 base 坐标系速度恢复世界坐标系速度时，简单的旋转矩阵变换 `R_b2w @ v_base` 是不完整的。正确的公式需要包含 Coriolis 项：

```
v_world = v_robot_world + R_b2w @ v_base + ω × r_world
```

其中 `ω × r_world` 是由于机器人旋转产生的牵连速度。在我们的测试中，Coriolis 修正使 TTC 速度误差从 0.97 m/s 降至 0.009 m/s，成功率从 5% 提升至 95%。

#### 洞察 3：Kalman 滤波器不适合高速球的速度估计

ObjectKalmanFilter 使用 6 维状态 [x,y,z,vx,vy,vz] 和匀加速运动模型（含重力）。在球速度 ~13 m/s 的情况下，默认参数 (process_noise=0.05, measurement_noise=0.08) 导致速度收敛时间常数过长。即使调整噪声参数（process_noise=1~100），速度估计误差仍保持在 15 m/s 左右。

根本原因：Kalman 滤波器的速度估计是对历史位置的加权平均，而位置差分速度直接反映瞬时速度变化。在 60Hz 更新频率下，位置差分已经足够平滑，不需要 Kalman 速度平滑。

#### 洞察 4：球检测率是 real 模式的首要瓶颈

球检测率从 <1%（960x540）提升到 65-80%（1920x1080 + min_area=3），real 模式成功率从 0% 提升到 10%。检测率每提升 10%，预计成功率可提升 5-8%。要达到 50%+ 成功率，需要将检测率提升到 90%+。
