# 第 1 章 为什么大多数 Skills 写不好

## 本章解决什么问题

大多数失败的 Skill 并不是“写得不够长”，而是没有被写成工程契约。本章先建立边界：Skill 不是普通 prompt，不是 tool schema，也不是完整 Agent。

## 核心概念

传统“模型 + 工具调用”有三个瓶颈：

- 上下文爆炸：把规范、工具、示例一次性塞给模型，成本高且干扰推理。
- 能力鸿沟：模型知道工具存在，却不知道领域专家如何把工具用好。
- 执行不可控：没有稳定流程、验证点和失败处理，同一任务多次执行结果漂移。

Skill 的价值是把可复用流程变成标准化程序性知识。

| 概念 | 核心定位 | 与 Skill 的关系 |
| --- | --- | --- |
| Prompt | 单次对话指令 | Skill 是可复用、可版本化的工程化 prompt 体系 |
| Tool | 原子外部能力 | Skill 规定何时、如何组合工具 |
| MCP | 外部工具和资源协议 | Skill 通过 MCP 使用外部能力 |
| Hook | 生命周期控制点 | Skill 的权限、审计、熔断可由 Hook 承担 |
| Plugin | 分发载体 | Skill 可随 Plugin 安装、升级、禁用 |
| Agent | 决策运行体 | Agent 选择并执行 Skill |

## 工程方法

判断一个流程是否适合做成 Skill，问四个问题：

1. 是否重复出现？
2. 是否有清晰输入和交付物？
3. 是否存在专家步骤或避坑经验？
4. 是否需要稳定输出、权限约束或回归测试？

## 反例

坏 Skill：“帮我处理各种代码问题”。  
问题：范围过大、触发模糊、输出不可验证、工具权限无法收敛。

好 Skill：“review-pr-risk”。  
价值：输入是 PR，输出是风险报告，步骤可固定，结果可回归。

## 练习

列出你团队 5 个重复任务，按候选评估表打分，选择最高分的一个作为后续章节贯穿案例。

## 检查清单

- [ ] Skill 有明确任务边界
- [ ] 有稳定输入和输出
- [ ] 包含专家流程而不只是背景知识
- [ ] 能测试触发和结果
- [ ] 能定义权限范围
